# UX Makeover + Full Hebrew Localization — Plan

Scope: the deployed web app only — the four pages generated by
`bridge_trainer/app/webapp.py` (`index.html`, `p.html`, `lead.html`,
`dashboard.html`) and the auth/data layer `bridge_trainer/web/bt-firebase.js`.
Out of scope: the CLI HTML report (`app/report.py`) and the legacy authored
publisher (`app/publish.py`) — both are dev-facing and English is fine there.

## 1. Where the app stands today (audit)

What already exists (the "redesign layer v2" that shipped earlier):

* `lang="he" dir="rtl"` on every page, LTR islands for bridge diagrams.
* Light/dark themes + manual override, 3-step text scaling, bottom nav,
  settings sheet, skip link, focus-visible styles.
* Hebrew: nav, settings, filters UI, verdict headlines, dashboard, auth gate,
  problem-type taxonomy (`TYPE_NAMES`), difficulty names, screen-reader card
  and call labels.

What accreted since and hurts UX:

* **Home page overload.** Stacked on one screen: scenario segmented control
  (bidding/lead) → MP/IMP mode cards (lead only) → collapsible facet filters
  (difficulty × type with cross-faceted counts) → CTA → stats card → session
  summary (injected above everything via `?summary=1`). First-time users see
  a wall of controls before the one button that matters.
* **Session flow is implicit.** A "session" is a localStorage blob; the
  summary is a card spliced into the home page; there's no end-of-run moment.
* **Verdict pages are dense.** p.html reveals headline + EV rows + win/push/
  loss bars + chips + 5 collapsibles; lead.html reveals headline + result
  lines + bars + prose + ranked 6-column table + 2 collapsibles. Statistical
  jargon (single-dummy, DD fog, CI) surfaces untranslated and unexplained.
* **Language is ~80/20 Hebrew/English**, with mixed-language lines that read
  poorly, and *all* engine-generated bid meanings in English.

Engine-generated content (the hard 20%):

* Bid meanings built by `terse()` from GIB convention cards: convention names
  ("Stayman", "Transfer to !H", "RKC Blackwood", "Lebensohl after double",
  "Balanced", "overcall", "weak", …) plus range fragments ("11-15", "5+♠").
  The numeric fragments are language-neutral; only the *name vocabulary* needs
  translation. This is a **closed, small set** (GIB's canonical names).
* `OPTSHOWS` fallback regexes that rewrite baked English prose
  ("limited — at most N HCP").
* Lead-page auction meanings use raw `explanations.auction[j].text` (English
  GIB prose) — unlike p.html which runs cards through `terse()`.

## 3. Phase plan

### Phase 0 — Localization foundation (enabler, no visual change)

1. **Central string table.** Extract every UI string into one `HE = {...}`
   dictionary at the top of `_SHARED_JS` (keys like `HE.selectAll`,
   `HE.startPractice`). Pages reference keys; no more string literals inline.
   This turns the remaining translation work — and every future feature —
   into a one-file edit and makes an "English leak" test possible.
2. **Readable source.** Replace all `א…`-escaped Hebrew in webapp.py
   with literal UTF-8 Hebrew (the file already mixes both; literals win for
   maintainability).
3. **Bridge-Hebrew glossary module** (`HE.gloss`): canonical Hebrew for the
   domain vocabulary, used everywhere consistently:
   לקיחות (not "טריקים" — fix existing dashboard/lead strings), הכרזה, הובלה,
   מכרז, חוזה, שליט / ללא שליט, מוכפל, פגיעות, מחלק, סדרה, נק"ג (HCP),
   כשל/הכשלה (set), סוויング→"הפרש" (swing), רווח־סמך (CI).
4. **GIB convention-name map** (`HE.conv`): translate the closed set of names
   `terse()` emits — סטיימן, העברה (טרנספר), בלאקווד RKC, קיו־ביד, לבנסול,
   אוברקול/התערבות, חלש, מזמין, מאזנת… Unmapped names fall back to English
   *and* `console.info` so the map converges over real pool data. Also route
   the lead page's auction notes through `terse()` + this map instead of raw
   English prose.

### Phase 1 — Complete the translation (string sweep)

Apply the §2 inventory end-to-end: chrome strings, aria labels, tooltips,
titles, vulnerability names, seat roles, chips, footnotes, fallbacks.
Acceptance: an automated test renders all four pages and asserts no
blacklisted English words appear outside code/`<script>` internals (see
Phase 6); a manual pass over one bidding problem, one lead problem (MP and
IMP), dashboard, settings, auth gate, and every empty/error state.

### Phase 2 — Information architecture: home + session flow

1. **Home = one decision.** Replace segmented-control + hidden mode card with
   two large scenario cards ("תרגול הכרזה" / "תרגול הובלה"), each carrying its
   own count of waiting problems; the lead card holds the MP/IMP toggle
   inline as two small pills. One primary CTA per card. Filters stay
   collapsed below as today, restyled as "התאמה אישית".
2. **First-run clarity.** For a signed-in user with 0 attempts, show a short
   one-card explainer (מה זה האימון, איך מנוקד) instead of the stats card.
3. **Real session loop.** Keep the 10-problem run, but give it a real finish:
   a dedicated summary view (accuracy, per-type breakdown of the run, "עוד
   סבב" / "לדף הבית" / "לתרגל את הטעויות") rendered as its own state instead
   of a query-param card on the home page. Session ribbon stays.
4. **Consistent app bar.** One shared top bar component (back link, page
   title, contextual meta) across p/lead/dashboard — today each page
   hand-rolls a slightly different `topbar`.

### Phase 3 — Problem pages (bidding + lead) hierarchy

1. **Answer-first layout** (both pages): type badge → auction → hand →
   answer controls, with everything else (mode banner details, hints line)
   demoted to compact strips. On lead.html merge the meta line and mode
   banner into one compact contract strip: `‎4♠ מוכפל · מכריז מזרח · פגיעות:
   כולם · IMP‎`.
2. **Verdict redesign, learn-first:**
   * Headline (נכון/עדיף היה) + one-sentence takeaway stay on top.
   * A single "למה?" block: the chosen vs best option comparison only
     (EV/לקיחות gap, one bar), in plain Hebrew.
   * Everything else — full option table, all-13-leads table, chips,
     raw DD, full deal, bid-by-bid review — moves into a uniform accordion
     ("פירוט מלא"), same order and styling on both pages.
   * Jargon gets inline info-tooltips in Hebrew (מהו single-dummy, מהו ערפל
     DD, מהו רווח־סמך) instead of untranslated terms.
3. **Confirm step polish.** Keep two-tap confirm; make the confirm card a
   sticky bottom sheet above the nav so the "הכרז ‎3♠‎" button never sits
   below the fold on small phones.
4. **Loading skeletons** for problem fetch (card-shaped shimmer) instead of a
   blank page; designed error states already exist — reuse.

### Phase 4 — Dashboard makeover

1. Restructure into three tabs/sections: סקירה (stat grid, trend, weak-area
   CTA) · הכרזה · הובלה (MP/IMP sub-sections). Today it's one long scroll.
2. Weak-area card ("מה כדאי לתרגל") moves to the top with its practice CTA.
3. Recent misses list gains the problem's type badge + difficulty stars, and
   the whole row becomes tappable.
4. Trend chart: axis labels in Hebrew, last-N window besides cumulative
   (cumulative flattens and hides recent improvement), keep SVG (no deps).

### Phase 5 — Visual polish

1. **Hebrew-first typography:** self-hosted/system Hebrew stack
   (`"Segoe UI", Heebo, Assistant, system-ui…` — decide on webfont vs system,
   §7); bump line-height for Hebrew; audit `letter-spacing` rules (uppercase
   tracking styles don't apply to Hebrew and currently produce odd spacing on
   labels like the type badge).
2. **Directionality audit:** every composite number/notation line
   (`+1.3 ±0.4 IMP`, `4♠×`, `61%`) wrapped as an LTR isolate; verify no
   mirrored parentheses/mis-ordered "·" separators in mixed lines (several
   exist today on subline/meta lines).
3. Real SVG icons for the bottom nav (♠/▤/⚙ glyphs render inconsistently
   across Android/iOS), consistent radii/shadows, felt background and BBO
   four-color deck stay — they're the brand.
4. Micro-feedback on reveal: brief highlight animation on the verdict card,
   ✓/✗ state on the chosen button (exists) — respecting
   `prefers-reduced-motion` (pattern already established).

### Phase 6 — QA, accessibility, tests, rollout

1. **No-English regression test** (pytest): render the four pages via
   `write_app()`, strip `<script>`/`<style>`, assert no `[A-Za-z]` runs
   except an allowlist (brand name, "IMP", "MP", seat letters N/E/S/W, card
   ranks, "NT", "X/XX"). This locks Phase 1 in forever.
2. **String-table discipline test:** grep generated JS for Hebrew literals
   outside the `HE` table (soft check) so new features go through the table.
3. **Playwright smoke** (Chromium is preinstalled): load each page with a
   stubbed `window.BT` fixture (no Firestore), answer one bidding + one lead
   problem, screenshot light/dark × RTL, assert axe-core has no criticals.
   Firestore stubbing keeps CI hermetic.
4. **A11y pass:** Hebrew aria labels everywhere (most exist), focus order on
   verdict reveal (exists — keep), contrast re-check of the four suit colors
   and gold CTA on both themes.
5. **Rollout:** one PR per phase (0+1 can ship together), each independently
   deployable; publish workflow (`trainer webapp` output) is untouched by
   all phases — pure content changes to webapp.py + bt-firebase.js.

## 4. Explicitly out of scope

* `app/report.py` CLI reports and `app/publish.py` legacy site (dev-only).
* Translating *stored pool data* (Firestore docs keep English `text`; we
  translate at render time — no data migration, old problems benefit too).
* Bidding-system/engine behavior, scoring, pool generation.

## 5. Sizing

| Phase | Size | Risk |
|---|---|---|
| 0 foundation | M | low — mechanical extraction, no behavior change |
| 1 translation sweep | M | low — string-only; glossary choices need review |
| 2 home/session IA | M | medium — touches session state machine |
| 3 problem pages | L | medium — verdict layout rework on two pages |
| 4 dashboard | M | low |
| 5 visual polish | S–M | low |
| 6 QA/tests | M | low; Playwright fixture is the main new piece |

## 6. Suggested order

0 → 1 ship together (immediate visible win: fully-Hebrew app), then 2 → 3 →
4 → 5, with 6's no-English test landing with 1 and the Playwright harness
landing before 3 (it de-risks the layout rework).

## 7. Open decisions (need product input)

1. **Brand:** keep "Bridge Trainer" in Latin (common in Israeli bridge apps)
   or go "מאמן הברידג'"? Recommendation: Hebrew title, Latin wordmark kept in
   the auth gate/logo.
2. **Call labels:** פס/כפל/כפל חוזר vs international Pass/Dbl/Rdbl.
   Recommendation: Hebrew — Israeli club players say פס/כפל; suit bids stay
   symbolic (3♠) either way.
3. **Convention names:** translate (סטיימן, העברה) vs keep English.
   Recommendation: translate the famous ones, keep rare ones English via the
   fallback path.
4. **MP/IMP naming:** keep Latin "MP"/"IMP" chips (universal scoring jargon)
   with Hebrew descriptions. Recommendation: yes — translating these hurts
   recognition.
5. **"לקיחות" vs "טריקים":** recommendation לקיחות (standard Israeli
   terminology); currently the app says טריקים in several places.
6. **Webfont:** ship Heebo/Assistant (adds a request on GitHub Pages, better
   Hebrew rendering) vs system stack (zero-cost). Recommendation: system
   stack first, webfont as a Phase-5 experiment.
